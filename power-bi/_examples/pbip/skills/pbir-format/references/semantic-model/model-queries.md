# Querying a semantic model with DAX

Query semantic models to validate fields, test measures, investigate values, or debug performance issues.

## Workflow

1. **Understand the objective** - What do you need to query for?
   - Validate field values for filters
   - Test measure calculations
   - Debug unexpected visual results
   - Explore available data

2. **Identify the model** - Find definition.pbir to determine workspace and model

   ```bash
   cat tmp/Test/Test.Report/definition.pbir
   # Look for: workspace name in connectionString (myorg/<workspace>)
   # Look for: model ID in semanticmodelid=<guid>
   ```

3. **Identify fields** - Determine what tables/columns/measures exist
   - Check model TMDL files in `tmp/models/<model-name>/definition/tables/`
   - Or query for available values: `EVALUATE VALUES('Table'[Column])`

4. **Compose the query** - Write DAX query using identified fields

   ```dax
   EVALUATE SUMMARIZECOLUMNS(
       'Table'[GroupingColumn],
       "Measure", Table[Measure]
   )
   ```

5. **Execute and interpret** - Run query and use results to solve the problem

   ```bash
   python3 scripts/query-model.py tmp/Test/Test.Report "<query>"
   ```

## Methods

### 1. query-model.py script (Recommended for PBIR reports)

Query the semantic model referenced by a PBIR report file.

**Usage:**

```bash
python3 scripts/query-model.py <report-path> <dax-query>
```

**How it works:**

1. Reads `definition.pbir` to extract workspace and model ID
2. Uses `fab api` to execute query via Power BI REST API
3. Returns JSON results

**Examples:**

```bash
# Get distinct values from a column
python3 scripts/query-model.py tmp/Test/Test.Report \
  "EVALUATE TOPN(10, VALUES('Products'[Type]))"

# Aggregate measure by dimension
python3 scripts/query-model.py tmp/Test/Test.Report \
  "EVALUATE SUMMARIZECOLUMNS('Date'[Calendar Year (ie 2021)], \"Order Lines\", 'Orders'[Order Lines])"

# Multiple measures
python3 scripts/query-model.py tmp/Test/Test.Report \
  "EVALUATE SUMMARIZECOLUMNS('Date'[Calendar Year (ie 2021)], \"Order Lines\", 'Orders'[Order Lines], \"Turnover\", 'Invoices'[Turnover])"
```

**Output:**

```json
{
  "status_code": 200,
  "text": {
    "results": [
      {
        "tables": [
          {
            "rows": [
              {
                "Date[Calendar Year (ie 2021)]": "2021",
                "[Order Lines]": 155548,
                "[Turnover]": 45413788565.3991
              }
            ]
          }
        ]
      }
    ]
  }
}
```

**Requirements:**

- Report must use service-hosted semantic model (`byConnection` in definition.pbir)
- `fab` CLI must be configured with authentication

**Limitations:**

- Cannot query reports with local models (`byPath`)
- DMV queries (INFO.MEASURES, INFO.COLUMNS) may not work in service

### 2. DAX Studio (Full-featured querying)

Best for interactive exploration and performance analysis.

1. Connect to semantic model in Power BI Service
2. Write and execute DAX queries
3. View results, query plans, and performance metrics

**Advantages:**

- Full DAX IntelliSense
- Query performance metrics
- Export results to Excel/CSV
- View execution plans
- DMV queries supported (when connected to local/SSAS models)

### 3. Performance Analyzer (Power BI Desktop)

View actual queries generated by visuals.

1. View → Performance Analyzer → Start Recording
2. Interact with visual (click, filter, etc.)
3. Click visual in Performance Analyzer → Copy query

**Use case:** See exactly what query a visual generates to understand or debug it.

### 4. Fabric CLI direct query

Query any semantic model in Fabric using `fab` CLI.

```bash
# Get workspace ID
WORKSPACE_ID=$(fab get "MyWorkspace.Workspace" -q "id" | tr -d '"')

# Get model ID
MODEL_ID=$(fab get "MyWorkspace.Workspace/MyModel.SemanticModel" -q "id" | tr -d '"')

# Execute query
fab api -A powerbi -X post \
  "groups/$WORKSPACE_ID/datasets/$MODEL_ID/executeQueries" \
  -i '{"queries":[{"query":"EVALUATE TOPN(10, '\''Table'\'')"}],"serializerSettings":{"includeNulls":true}}'
```

## Common query patterns

### Get distinct column values

```dax
EVALUATE VALUES('Table'[Column])
ORDER BY 'Table'[Column]
```

### Get top N rows

```dax
EVALUATE TOPN(10, 'Table')
ORDER BY 'Table'[Column] DESC
```

### Aggregate measures

```dax
EVALUATE
SUMMARIZECOLUMNS(
    'Dimension'[Column],
    "MeasureAlias", Table[Measure]
)
```

### Multiple grouping columns

```dax
EVALUATE
SUMMARIZECOLUMNS(
    'Date'[Year],
    'Products'[Category],
    "Revenue", [Total Revenue],
    "Units", [Total Units]
)
```

### Single measure value (card-style)

```dax
EVALUATE ROW("Revenue", [Total Revenue])
```

Or the scalar shorthand:

```dax
EVALUATE {[Total Revenue]}
```

**Important:** `SUMMARIZECOLUMNS` with only `IGNORE()`-tagged expressions and no grouping columns returns an **empty table** — this is wrong for card-style queries. Use `EVALUATE ROW()` or `EVALUATE {measure}` instead. `IGNORE()` only applies when grouping columns are present and a measure might return blank for some groups.

## Use cases

### 1. Validate field values for filters

Before setting filter values in visual metadata, check what values actually exist:

```bash
python3 scripts/query-model.py tmp/Test/Test.Report \
  "EVALUATE VALUES('Date'[Calendar Year (ie 2021)])"
```

### 2. Test measure expressions

Verify a measure returns expected results before using it in a visual:

```bash
python3 scripts/query-model.py tmp/Test/Test.Report \
  "EVALUATE SUMMARIZECOLUMNS(\"Test\", 'Orders'[Order Lines])"
```

### 3. Debug unexpected visual values

Run the same query a visual generates to see if the issue is the query or visual rendering:

```bash
# Get query from Performance Analyzer, then run it
python3 scripts/query-model.py tmp/Test/Test.Report \
  "EVALUATE SUMMARIZECOLUMNS('Date'[Month], \"Lines\", 'Orders'[Order Lines])"
```

### 4. Check measure dependencies

Before creating an extension measure that references model measures, verify they exist and work:

```bash
python3 scripts/query-model.py tmp/Test/Test.Report \
  "EVALUATE SUMMARIZECOLUMNS(\"Budget\", 'Budget'[Total Budget])"
```

## Inferring queries from visuals

See [Inferring queries from visuals](inferring-queries-from-visuals.md) for how to map visual metadata to DAX queries.

Quick summary:

- Visual projections → SUMMARIZECOLUMNS arguments
- Column in Category role → Grouping column
- Measure in Y/Values role → Measure argument
- Extension measures (Schema: "extension") → DEFINE section

## Troubleshooting

### "Could not extract model ID from connection string"

The report uses a local model (byPath). Use DAX Studio to connect to the .pbix file instead.

### "Failed to execute the DAX query" (400 error)

Common causes:

- Invalid DAX syntax
- Referencing non-existent tables/columns/measures
- Ambiguous column references (need table prefix)
- DMV queries not supported in service (INFO.MEASURES, etc.)

### DMV queries (INFO.* functions) fail in Power BI Service

**Problem:**

```bash
python3 scripts/query-model.py tmp/Test/Test.Report \
  "EVALUATE TOPN(5, SELECTCOLUMNS(INFO.MEASURES(), \"Table\", [TableName], \"Measure\", [MeasureName]))"

# Returns: DatasetExecuteQueriesError (400)
```

**Why:** DMV (Dynamic Management Views) queries like `INFO.MEASURES()`, `INFO.COLUMNS()`, `INFO.TABLES()` only work when connected to:
- Analysis Services (on-prem)
- Power BI Desktop local models
- Azure Analysis Services

They do **not** work against Power BI Service semantic models via the REST API.

**Workaround:** To get model metadata:

1. **Use TMDL files** - Model definition in `tmp/models/<model-name>/definition/tables/`
   ```bash
   # List all measures
   for file in tmp/models/spaceparts-otc-full/definition/tables/*.tmdl; do
     grep "^\tmeasure " "$file"
   done
   ```

2. **Use DAX Studio** - Connect directly to workspace/model
   - Supports full DMV queries
   - Better for metadata exploration

3. **Query actual data** - Instead of metadata, query actual values:
   ```dax
   -- Instead of INFO.COLUMNS() for Date table
   EVALUATE TOPN(1, 'Date')
   -- See all column names in results
   ```

### TREATAS filter context errors

**Problem:**

```bash
python3 scripts/query-model.py tmp/Test/Test.Report \
  "EVALUATE SUMMARIZECOLUMNS(TREATAS({\"2021\"}, 'Date'[Year]), 'Date'[Quarter], \"Lines\", [Order Lines])"

# Error: "A single value for column 'Quarter' in table 'Date' cannot be determined"
```

**Why:** When using TREATAS as a filter, all grouping columns must have proper filter context. The filter on Year doesn't automatically filter Quarter values.

**Fix:** Use TREATAS in filter position, not grouping position:

```dax
-- Wrong
EVALUATE SUMMARIZECOLUMNS(
    TREATAS({"2021"}, 'Date'[Year]),  -- Filter in grouping position causes issues
    'Date'[Quarter],
    "Lines", [Order Lines]
)

-- Right
EVALUATE SUMMARIZECOLUMNS(
    'Date'[Quarter],
    TREATAS({"2021"}, 'Date'[Year]),  -- Filter after grouping columns
    "Lines", [Order Lines]
)
```

Or filter in CALCULATETABLE:

```dax
-- Without KEEPFILTERS (overrides existing filters)
EVALUATE CALCULATETABLE(
    SUMMARIZECOLUMNS(
        'Date'[Quarter],
        "Lines", [Order Lines]
    ),
    'Date'[Year] = "2021"  -- Replaces any existing Year filter
)

-- With KEEPFILTERS (respects existing filters)
EVALUATE CALCULATETABLE(
    SUMMARIZECOLUMNS(
        'Date'[Quarter],
        "Lines", [Order Lines]
    ),
    KEEPFILTERS('Date'[Year] = "2021")  -- Intersects with existing filters
)
```

**KEEPFILTERS usage:**

- **Without KEEPFILTERS:** Filter replaces existing context (ignores slicer selections, page filters)
- **With KEEPFILTERS:** Filter intersects with existing context (respects user selections)

Use KEEPFILTERS when:
- Query should respect report/page filter context
- Working with slicers that might already filter the same column
- Need to preserve existing user selections

Don't use KEEPFILTERS when:
- Explicitly overriding filters (e.g., showing YTD regardless of date slicer)
- Testing queries in isolation without filter context

### Query times out

- Reduce TOPN limit
- Remove unnecessary grouping columns
- Filter to smaller date range
- Check measure performance in DAX Studio
