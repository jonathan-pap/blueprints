# Selectors — when/what a formatting property applies to

> A property inside a visual's `objects` (or `visualContainerObjects`) can carry a **selector**
> that scopes it: globally, to one series, to one data point, to a category value, or to a
> visual-element state. Get the selector wrong and conditional formatting silently misfires.
> Source schema: `formattingObjectDefinitions/1.5.0`.

## Selector shape

```json
"selector": {
  "data": [...],          // scope by data (per-point / category / role / total)
  "metadata": "...",      // scope by field (Table.Measure, or "select" for visual calcs)
  "id": "...",            // scope by element state (default / hover / selected / press)
  "highlightMatching": 0, // optional cross-filter behavior
  "hierarchyMatching": 0, // optional matrix-hierarchy behavior
  "order": 0              // optional tie-breaker when selectors overlap
}
```

## The five selector types

| Type | Scopes to | Use when |
|---|---|---|
| **none** | whole visual element | one title/legend/background for the visual |
| **metadata** | one series/field | color the "Revenue" series; target a visual calc |
| **data → dataViewWildcard** | every point (or totals) | per-point conditional formatting (the common case) |
| **data → scopeId** | one category value | "Electronics" bar a different color |
| **data → roles** | every field in a role | all Category fields |
| **id** | element state | slicer selected item, button hover |

## `dataViewWildcard` — the conditional-formatting workhorse

```json
"selector": { "data": [{ "dataViewWildcard": { "matchingOption": 1 } }] }
```

| matchingOption | Matches | Use for |
|---|---|---|
| **0** | identities + totals | series-level / identity-matched color; **wrong for per-point** |
| **1** | instances only | **per-point** conditional formatting (most common) |
| **2** | totals only | total-row formatting |

This is the single most important pattern in the file: `matchingOption: 1` is what makes a
measure-driven color evaluate **per data point**.

## The two-entry array pattern (required for measure-driven formatting)

Conditional formatting needs **two** entries in the property array: a base entry with no
selector, then a conditional entry with the `dataViewWildcard` selector.

```json
"dataPoint": [
  { "properties": {} },                                   // base / fallback
  {
    "properties": {
      "fill": { "solid": { "color": { "expr": { "Measure": {
        "Expression": { "SourceRef": { "Schema": "extension", "Entity": "_Formatting" } },
        "Property": "PosNegColor"
      } } } } }
    },
    "selector": { "data": [{ "dataViewWildcard": { "matchingOption": 1 } }] }
  }
]
```

A single combined entry (selector + measure in one object) produces unpredictable rendering.
Always split base from conditional.

**Objects that support this pattern:** `dataPoint`, `lineStyles`, `error`.
**Objects that do NOT support selectors:** `title`, `legend`, `background`, `border`,
`categoryAxis`, `valueAxis` — a selector on these is silently ignored and the property applies
globally.

## `metadata` — target one series (or a visual calculation)

```json
"selector": { "metadata": "Sales.Revenue" }   // color just the Revenue series
```

- Model/extension measure: `"metadata": "Table.Measure"`.
- **Visual calculation:** `"metadata": "select"` — every `NativeVisualCalculation` carries
  `queryRef: "select"`, so all visual calcs share this selector (you cannot distinguish two
  visual calcs on the same visual). This is the "spotlight the latest point" trick: hide
  markers/labels on the base measure, show them on `metadata: "select"`.

See [../calculations/visual-calculation.md](../calculations/visual-calculation.md).

## `scopeId` — one specific category value

```json
"selector": { "data": [{ "scopeId": { "Comparison": {
  "ComparisonKind": 0,
  "Left": { "Column": { "Expression": { "SourceRef": { "Entity": "Products" } }, "Property": "Category" } },
  "Right": { "Literal": { "Value": "'Electronics'" } }
} } }] }
```

Applies only where `Category = "Electronics"`. Verbose and usually generated by the Desktop UI.
Note: `scopeId` uses `SourceRef.Entity` directly — unlike filter `Where` conditions, which use
a `Source` alias declared in a `From[]` array (see [expressions.md](expressions.md#sourceref-context)).

## `roles` — every field in a role

```json
"selector": { "data": [{ "roles": ["Category"] }] }
```

Common roles: `Category`, `Values`, `Series`, `X`, `Y`, `Legend`, `Tooltips`.

## `id` — element states (slicers, buttons, shapes)

| id | State |
|---|---|
| `"default"` | base / unselected |
| `"selection:selected"` | selected/active item |
| `"interaction:hover"` | mouse hover |
| `"interaction:press"` | press/click active |

```json
"items": [
  { "properties": { "fontColor": { "solid": { "color": { "expr": { "Literal": { "Value": "'#000000'" } } } } } },
    "selector": { "id": "default" } },
  { "properties": { "fontColor": { "solid": { "color": { "expr": { "Literal": { "Value": "'#FFFFFF'" } } } } } },
    "selector": { "id": "selection:selected" } }
]
```

Slicers and action buttons have the richest state set; most other visuals support only `"default"`.

## Decision table

| Goal | Selector | matchingOption |
|---|---|---|
| Format the whole visual element | none | — |
| Color one specific series | `metadata: "Table.Measure"` | — |
| Format the visual-calc series | `metadata: "select"` | — |
| Per-point measure colors | `data` + `dataViewWildcard` | **1** |
| Series-level / identity color | `data` + `dataViewWildcard` | 0 |
| Format one category value | `data` + `scopeId` | — |
| Format all fields in a role | `data` + `roles` | — |
| Format totals only | `data` + `dataViewWildcard` | 2 |

## Top mistakes

1. **`matchingOption: 0` when you wanted per-point** → all points get one color (series behavior).
2. **Single-entry conditional array** → drop the base `{ "properties": {} }` entry and rendering is unpredictable.
3. **Selector on `title`/`legend`/`background`/`border`** → ignored; the property applies globally anyway.
